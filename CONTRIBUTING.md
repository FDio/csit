# Contributing to this Project

First off, thank you for considering contributing! We value all
contributions, whether they are bug fixes, new features, documentation
updates, or architectural improvements.

To ensure the legal integrity of this project and protect both our
contributors and downstream users, we have established guidelines for
licensing, tracking the origin of code, and handling non-copyrightable text.

## 1. Licensing Overview

This project uses the open-source licenses detailed in the LICENSE file
located at the root of this repository.

We track licensing at the file level using SPDX License Identifiers
([https://spdx.dev/ids/](https://spdx.dev/ids/)). Files containing human-authored, copyrightable
material must include the appropriate SPDX header at the top of the file
corresponding to the licenses outlined in the LICENSE file.

Because contributors and source material may originate in different
jurisdictions, copyright characterization can vary. This document states the
project's contribution policy for origin tracking and DCO interpretation, not a
complete summary of every applicable copyright law.

## 2. Developer Certificate of Origin (DCO)

We require all contributions to comply with the Developer Certificate of
Origin (DCO). You can read the full text at
[https://developercertificate.org](https://developercertificate.org).

To certify your compliance, every commit must include a Signed-off-by line.
This line must be the final trailer and the final line of the commit message
when the contribution is submitted. If any workflow appends content after that
line, the submitted contribution has changed and a human contributor must add a
new final Signed-off-by line certifying the updated state.

Requirements for Signed-off-by:

* The signing entity must be a human person. AI agents, bots, and
automated scripts cannot legally certify the DCO.
* You must use a valid personal or corporate e-mail address. Group
aliases or team mailing lists are not permitted.
* By adding this line, you affirm that you have the right to submit the text
under the project's licenses.

Note on "Creation": The DCO references contributions "created" by you. If you
incorporate pure AI-generated text or existing Public Domain material, you
did not create that text. However, because nobody owns Public Domain material,
you still have the legal right to distribute it. You must declare this
origin in the commit message so the project history remains accurate.

Example:
```
Signed-off-by: Vratko Polak <vrpolak@cisco.com>
```

## 3. Core Concepts

To follow these guidelines, it is helpful to understand three core concepts.
We define them strictly as follows:

* Human-Written Code: The lines of text typed by a human to implement logic.
Writing a prompt is human-written text, but the code an AI generates in
response is not human-written code.
* Copyright Holder: The person or company that legally owns the human-written
code. Under typical employment contracts, the employer is automatically the
Copyright Holder of your human-written code.
* Public Domain Code: Code that the project treats as having no Copyright
Holder for contribution-tracking purposes (such as text generated by an AI, or
work prepared by a U.S. Government employee as part of official duties). Anyone
has the right to distribute it, but no one owns it.

## 4. Default Authorship and Disclaimers

For a typical commit containing only human-written code, signing the DCO means
you certify that you have the right to submit that code under the project's
licenses. In that common case, the copyright holder is usually the human author
or, under typical employment contracts, the author's employer.

If your commit includes code where you are not the sole human creator (such as
Generative AI outputs or Public Domain snippets), you must explicitly state
this in the Git commit message.

We do not enforce a strict formatting syntax for this disclaimer. You may use
natural language narrative or community-standard Git trailers (such as
Assisted-by: or Co-authored-by:). The only requirement is that a subsequent
reviewer or auditor reading the commit message can easily understand which
parts of the diff are your copyrighted additions, and which parts originate
from an uncopyrightable public source. If the commit message does not clearly
identify the origin of non-human or Public Domain text, the project treats the
Signed-off-by line as a certification that the contributor is submitting the
entire commit under the normal human-authored copyright path.

## 5. Examples of Tracking Scenarios

The following examples illustrate how these concepts apply to daily code
contributions and repository management.

Example 1: Manual but Insubstantial Refactor

* Scenario: A developer opens an existing, fully copyrighted file and spends
an hour manually adjusting line lengths, fixing indentation, and changing
single-letter loop indexes to descriptive local names. No new logic,
architecture, explanatory comments, or domain-specific naming is introduced.
* Analysis: Even though this work was done entirely by a human without tools,
purely mechanical or insubstantial edits usually do not change the
origin-tracking status of the file. When an edit plausibly adds original
expression, such as creative API names, explanatory comments, meaningful
restructuring, or domain-specific naming, disclose that portion as
Human-Written Code.

Example 2: AI Draft with Human Fixes

* Scenario: A developer uses an AI tool to generate a complex data parsing
function, but the AI output contains a bug. The developer manually rewrites
the error handling and the API calls to make it work.
* Analysis: The initial text generated by the AI is Public Domain. However,
the manual bug fixes represent creative Human-Written Code. The employer
owns the copyright to those specific fixes. The human-authored portions are
licensed under the project's license, while the Public Domain portions remain
Public Domain.

Example 3: The Blended Commit (Gerrit Patchsets)

* Scenario: A developer uses AI to generate a baseline feature. The project's
CI tests fail. The developer manually writes original code to fix the logic.
Because the code review system requires a single, passing commit to merge,
the AI code and the human fixes must be combined into one patchset.
* Analysis: This unified commit contains both Public Domain code and creative
Human-Written Code. While copyright law automatically protects the human
fixes, this project document explicitly requires the contributor to describe
the boundary within the commit message so the repository maintains an
honest legal history.

## 6. Sample Commit Messages

Below are examples of how to structure commit messages for blended or
non-copyrightable contributions.

### Sample Message A (Reflecting Example 2 and 3)

```
Add fast-path JSON parser for telemetry data

The initial boilerplate and the core tokenization loop were drafted
using generative AI.

I manually rewrote the handle_malformed_input function and the memory
allocation logic to fix a segfault and align with our internal standards.

Change-Id: Ia0b6820464249db5237cc3334d3a6ffb0bbf2344
Assisted-by: Claude:claude-3-opus
Signed-off-by: Alex Chen <alex.chen@example.com>
```

### Sample Message B (Public Domain Origin)

```
Implement coordinate conversion reference module

Added the coordinate_conversion.c reference implementation. This material was
originally published by a United States Geological Survey employee as part of
their official duties. It is a United States Government work and is not subject
to U.S. copyright protection.

The wrapper classes and associated unit tests are original human-authored
additions.

Change-Id: I234567890abcdef1234567890abcdef123456789
Signed-off-by: Jamie Rivera <j.rivera@example.com>
```
